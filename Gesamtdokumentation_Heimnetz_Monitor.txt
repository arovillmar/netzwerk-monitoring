────────────────────────────────────────────────────────────────────────────────
  MODUL 1 – Check-Ping.ps1
────────────────────────────────────────────────────────────────────────────────

  Funktion:    Check-Ping
  Zweck:       Grundlegender Erreichbarkeitstest jedes Geräts per ICMP-Ping.
               Erster Check für alle Geräte – schlägt er fehl, werden
               weiterführende Checks meist übersprungen.

  Parameter:
    $IP          IP-Adresse des Zielgeräts
    $TimeoutMs   Wartezeit in Millisekunden (Standard: 1000ms)

  Technik:     Test-Connection (ICMP Echo Request / Ping)

  Rückgabe:
    Status       "OK" oder "FEHLER"
    Latenz_ms    Antwortzeit in Millisekunden
    Info         Beschreibungstext

  Besonderheit: Misst die Latenz und gibt sie für die Konsolenanzeige zurück.
                LAN-Geräte: 0–2ms, WLAN: 3–10ms, Powerline: bis 700ms möglich.

  Verwendet für: Alle 17 Geräte als Basis-Check.


────────────────────────────────────────────────────────────────────────────────
  MODUL 2 – Check-Port.ps1
────────────────────────────────────────────────────────────────────────────────

  Funktion:    Check-Port
  Zweck:       Prüft ob ein bestimmter TCP-Port auf einem Gerät geöffnet
               und erreichbar ist. Universell einsetzbar für alle Port-Checks.

  Parameter:
    $IP          IP-Adresse
    $Port        TCP-Port (z.B. 80, 443, 3389, 8463)
    $TimeoutMs   Wartezeit (Standard: 2000ms)

  Technik:     Asynchroner TCP-Connect (System.Net.Sockets.TcpClient)
               BeginConnect + AsyncWaitHandle.WaitOne für nicht-blockierenden Test.

  Rückgabe:
    Status       "OFFEN" oder "GESCHLOSSEN"
    Port         geprüfter Port
    Zeit_ms      Verbindungszeit

  Verwendet für: Fritz!Box (80), SASCHA_SERVER (3389), Mailstore (8463),
                 Reolink-Kameras (554, 80, 9000), INSTAR (554, 8080)


────────────────────────────────────────────────────────────────────────────────
  MODUL 3 – Check-SSH.ps1
────────────────────────────────────────────────────────────────────────────────

  Funktion:    Check-SSH
  Zweck:       Verbindet sich per SSH auf Linux-Geräte, führt Systembefehle aus
               und wertet Uptime, Festplattennutzung und RAM-Status aus.

  Parameter:
    $IP          IP-Adresse
    $Port        SSH-Port (Standard: 22, Synology: 822)
    $User        SSH-Benutzername

  SSH-Befehl:  "uptime && df -h / && free -m"
               Liefert: Systemlaufzeit, Root-Partition-Auslastung, RAM-Belegung

  Technik:     ssh.exe (OpenSSH, in Windows 10/11 integriert)
               Key-Authentifizierung (kein Passwort im Code)
               Start-Job für echtes 10-Sekunden-Timeout

  SSH-Optionen:
    BatchMode=yes           Kein interaktiver Passwortdialog
    ConnectTimeout=5        5 Sekunden Verbindungsaufbau-Timeout
    StrictHostKeyChecking=no  Automatische Host-Key-Akzeptanz

  Rückgabe:
    Status       "OK" oder "FEHLER"
    Uptime       Systemlaufzeit-String
    Disk_Info    Root-Partition-Auslastung
    RAM_Info     RAM-Belegung (Mem: total used free)
    Info         Zusammenfassung

  Verwendet für: Raspberry Pi 5 (Port 22, User: pi)


────────────────────────────────────────────────────────────────────────────────
  MODUL 4 – Check-Camera.ps1
────────────────────────────────────────────────────────────────────────────────

  Funktion:    Check-Camera
  Zweck:       Vollständiger Kamera-Check mit fünf Prüfstufen:
               Ping → RTSP-Port → HTTP-Port → Stream-Analyse → API-Login

  Parameter:
    $IP                  IP-Adresse der Kamera
    $Typ                 "reolink" oder "instar"
    $HttpPort            HTTP-Port aus config.json (80, 8080 oder 9000)
    $FfprobePfad         Pfad zu ffprobe.exe (optional)
    $RtspUrl             Vollständige RTSP-URL (optional)
    $Name                Gerätename für Lockout-Tracking
    $ReolinkUser         API-Benutzername (Standard: "admin")
    $ReolinkPassSecure   Passwort als SecureString (optional)

  Prüfstufen:
    1. ICMP-Ping
    2. TCP-Port-Test auf RTSP/554
    3. TCP-Port-Test auf HTTP-Port (80, 8080 oder 9000)
    4. ffprobe Stream-Analyse (Auflösung, Codec) – nur wenn RTSP offen
    5. Reolink JSON-API: Login → GetDevInfo → Snapshot → Logout

  Technik:
    RTSP/HTTP:   System.Net.Sockets.TcpClient (asynchroner TCP-Test)
    ffprobe:     Externer Prozess via Start-Job (12s Timeout)
    Reolink API: Invoke-RestMethod (POST JSON, HTTPS mit SkipCertificateCheck)
    Snapshot:    Invoke-WebRequest (Bild-Download, Base64-kodiert im Rückgabeobjekt)

  Reolink API-Endpunkte (automatische Fallback-Suche):
    https://<IP>/api.cgi
    https://<IP>/cgi-bin/api.cgi

  Lockout-Schutz:
    Nach 3 fehlgeschlagenen API-Logins wird die Kamera-IP für 60 Minuten
    gesperrt. Verhindert Account-Sperrung durch den Kamera-eigenen Schutz.
    Speicherdatei: logs\camera_lockout.json

  Statuslogik:
    FEHLER   Kein Ping ODER weder RTSP noch HTTP erreichbar
    WARNUNG  Ping OK, aber RTSP oder HTTP fehlt
    OK       Ping + RTSP + HTTP erreichbar

  Verwendet für: Alle 5 Reolink-Kameras, INSTAR Kamera 1

  MODUL 7 – Check-SynologyAPI.ps1
────────────────────────────────────────────────────────────────────────────────

  Funktion:    Check-SynologyAPI
  Zweck:       Überwacht Synology NAS: Volume-Belegung, RAID-Status, Uptime.
               Warnt wenn ein Volume mehr als 80% belegt ist.

  Parameter:
    $IP          IP des NAS
    $Port        SSH-Port (Standard 5000, tatsächlich SSH-Port 822)
    $User        SSH-Benutzername (Armin)
    $PassSecure  (nicht mehr genutzt – Key-Auth)

  SSH-Befehl (4-in-1):
    "df -h 2>/dev/null | grep '/volume';
     cat /proc/mdstat 2>/dev/null | grep -E 'md[0-9]+\s*:';
     free -m 2>/dev/null | grep Mem;
     uptime 2>/dev/null"

  Auswertung:
    df -h        Festplattennutzung pro Volume (/volume1, /volume2 ...)
    /proc/mdstat Linux Software-RAID Status (md0, md1, md2 ...)
    free -m      RAM-Belegung
    uptime       Systemlaufzeit

  Filterregel: Nur Mounts die exakt auf /volume\d+$ passen werden gewertet
               (keine Sub-Mounts wie /volume1/@eaDir etc.)

  Technik:     SSH via ssh.exe (Start-Job, 12s Timeout)
               Key-Authentifizierung (SSH-Key muss eingerichtet sein)

  Rückgabe:
    Status, Volumes (Array mit Name/Groesse/Frei/Prozent), Uptime, Info

  Verwendet für: DS1525+ (192.168.80.206, Port 822), DS723+ (192.168.80.207)

  Hinweis: Ursprünglich über DSM REST-API (Port 5000) – umgestellt auf SSH,
           da keine Passwörter für die API-Authentifizierung nötig sind
           und die SSH-Key-Methode robuster ist.

  MODUL 8 – Check-MailstoreAPI.ps1
────────────────────────────────────────────────────────────────────────────────

  Funktion:    Check-MailstoreAPI
  Zweck:       Überwacht den MailStore Server auf der VM 192.168.80.120.
               Dreistufiger Ansatz: Ping → Windows-Dienst → API-Job-Auswertung

  Parameter:
    $IP            IP der Mailstore-VM (Standard: 192.168.80.120)
    $User          MailStore Admin-Benutzer (Standard: "admin")
    $PassSecure    MailStore Admin-Passwort als SecureString
    $WinUser       Windows-Benutzer für Remoting (Standard: "Administrator")
    $WinPassSecure Windows-Passwort für Remoting als SecureString (optional)
    $ApiPort       API-Port (Standard: 8463)

  STUFE 1 – Ping + RDP-Port:
    Test-Connection → VM erreichbar?
    TCP-Test Port 3389 → RDP-Port offen?
    Bei Ping-Fehler: Sofortrückgabe mit FEHLER

  STUFE 2 – PowerShell Remoting (optional):
    New-PSSession zum Windows-Host → Invoke-Command
    Get-Service → prüft ob MailStore-Dienst läuft
    Funktioniert nur wenn WinRM auf der VM aktiv ist
    Schlägt fehl → wird übersprungen, Check läuft trotzdem weiter

  STUFE 3 – MailStore Administration API (HTTP POST, Port 8463):
    Wichtig: API muss in "MailStore Server Service Configuration" aktiviert sein!

    GetServerInfo → API erreichbar? (Version, Machine-Name)
    GetJobs       → Job-Namen laden (für sprechende Ausgabe)
    GetJobResults → Ergebnisse letzter 24h (pro Job nur neuestes Ergebnis)

    HTTP Basic Auth: Base64(admin:passwort) im Authorization-Header
    Selbstsigniertes Zertifikat → SkipCertificateCheck = $true
    BOM-Behandlung: .TrimStart([char]0xFEFF) vor ConvertFrom-Json

  KRITISCH: Datumsformat für GetJobResults
    Korrekt:     yyyy-MM-ddTHH:mm:ss   (Beispiel: 2026-04-28T14:00:00)
    FALSCH:      yyyy-MM-dd HH:mm:ss   (Leerzeichen statt T)
    FALSCH:      dd.MM.yyyy HH:mm:ss   (deutsches Format)
    FALSCH:      MM/dd/yyyy HH:mm:ss   (US-Format)
    FALSCH:      ISO 8601 mit +02:00   (Timezone-Suffix nicht akzeptiert)

  GetJobResults Pflichtparameter:
    fromIncluding    Startzeit (yyyy-MM-ddTHH:mm:ss)
    toExcluding      Endzeit   (yyyy-MM-ddTHH:mm:ss)
    timeZoneId       Zeitzone  ("W. Europe Standard Time")

  Statuslogik:
    FEHLER   VM nicht erreichbar ODER Dienst gestoppt (wenn Remoting OK)
    WARNUNG  Mindestens ein Job mit result="failed" in den letzten 24h
    OK       Alle Jobs succeeded oder cancelled

  Rückgabe:
    Status, VM_Erreichbar, RDP_Offen, Remoting_OK, Dienst_Laeuft,
    API_Verfuegbar, Fehler_Jobs, Archiv_Gesamt, Jobs, Kritische_Jobs,
    Antwortzeit_ms, Info

  Verwendete API-Endpunkte:
    https://192.168.80.120:8463/api/invoke/GetServerInfo
    https://192.168.80.120:8463/api/invoke/GetJobs
    https://192.168.80.120:8463/api/invoke/GetJobResults
    https://192.168.80.120:8463/api/get-metadata   (nur für Dokumentation/Test)
    https://192.168.80.120:8463/api/invoke/GetTimeZones (nur für Test)

────────────────────────────────────────────────────────────────────────────────
  MODUL 14 – Check-EmailDelivery.ps1
────────────────────────────────────────────────────────────────────────────────

  Funktion:    Check-EmailDelivery
  Zweck:       Vollständiger E-Mail-Lieferungstest:
               TCP-Verbindung → SMTP-Authentifizierung → E-Mail versenden

  Parameter:
    $SmtpHost    SMTP-Server (exchange.ionos.eu)
    $SmtpPort    SMTP-Port (587)
    $Von         Absender-Adresse
    $An          Empfänger-Adresse
    $PassSecure  SMTP-Passwort als SecureString

  Prüfstufen:
    1. TCP-Test auf SMTP-Port 587
    2. SMTP-Login mit Credentials
    3. HTML-Test-E-Mail versenden

  Gesendete E-Mail:
    Betreff: "[Heimnetz Monitor] E-Mail Test <Datum> (ID: <TestID>)"

================================================================================
  TEIL C – VERBINDUNGS- UND ABFRAGETECHNIKEN
================================================================================

  Das System verwendet 9 verschiedene Verbindungstechniken:

────────────────────────────────────────────────────────────────────────────────
  TECHNIK 1 – ICMP PING (Test-Connection)
────────────────────────────────────────────────────────────────────────────────

  PowerShell-Cmdlet:  Test-Connection
  Protokoll:          ICMP (Internet Control Message Protocol)
  Funktion:           Sendet einen Echo-Request und wartet auf Echo-Reply

  Verwendung im Code:
    Test-Connection -ComputerName $IP -Count 1 -TimeoutSeconds 2 -ErrorAction Stop

  Vorteile:
    - Einfachster und schnellster Erreichbarkeitstest
    - Misst Latenz (Round-Trip-Time)
    - Kein offener Port nötig

  Nachteile:
    - ICMP kann durch Firewalls blockiert sein
    - Kein Protokolltest (nur IP-Schicht)

  Gibt zurück:        $ping.Latency (in Millisekunden)

  Genutzt von:        Alle 17 Module als Basisfunktion


────────────────────────────────────────────────────────────────────────────────
  TECHNIK 2 – ASYNCHRONER TCP-PORT-TEST
────────────────────────────────────────────────────────────────────────────────

  .NET-Klasse:        System.Net.Sockets.TcpClient
  Protokoll:          TCP

  Verwendung im Code:
    $client     = New-Object System.Net.Sockets.TcpClient
    $verbindung = $client.BeginConnect($IP, $Port, $null, $null)
    $erfolg     = $verbindung.AsyncWaitHandle.WaitOne($TimeoutMs, $false)
    if ($erfolg -and $client.Connected) { ... }
    $client.Close(); $client.Dispose()

  Warum asynchron?
    Der synchrone Connect() blockiert den PowerShell-Thread für die gesamte
    Timeout-Dauer. Mit BeginConnect + WaitOne wird nur so lange gewartet
    wie angegeben, dann wird abgebrochen – ohne den Thread zu blockieren.

  Genutzt für:
    Port 80 (HTTP), 443 (HTTPS), 554 (RTSP), 587 (SMTP), 822 (SSH Synology),
    3389 (RDP), 8463 (MailStore API), 9000 (Reolink Eingang HTTP)

  Genutzt von:        Check-Port, Check-Camera, Check-Powerline, Check-SMA,
                      Check-MailstoreAPI, Check-ExternalLogins, Check-EmailDelivery


────────────────────────────────────────────────────────────────────────────────
  TECHNIK 3 – SSH (Secure Shell mit Key-Authentifizierung)
────────────────────────────────────────────────────────────────────────────────

  Programm:           ssh.exe (Windows OpenSSH, integriert in Windows 10/11)
  Protokoll:          SSH (Secure Shell) – TCP Port 22 (Pi) oder 822 (Synology)
  Authentifizierung:  Public-Key (kein Passwort im Code!)

  SSH-Parameter:
    BatchMode=yes           Kein interaktiver Dialog
    ConnectTimeout=5        Verbindungsaufbau-Timeout in Sekunden
    StrictHostKeyChecking=no  Automatische Host-Key-Akzeptanz (Heimnetz)
    -p <Port>               Port-Angabe

  Timeout-Handling:   ssh.exe wird in Start-Job ausgeführt.
                      Wait-Job -Timeout 10/12 → bei Überschreitung: Stop-Job

  Code-Muster:
    $job = Start-Job -ScriptBlock {
        param($a)
        $out  = & ssh.exe @a 2>&1
        $code = $LASTEXITCODE
        [PSCustomObject]@{ Output = $out; ExitCode = $code }
    } -ArgumentList (,$sshArgs)
    $fertig = Wait-Job $job -Timeout 10

  Ausgeführte Linux-Befehle:
    Raspberry Pi:   uptime && df -h / && free -m
    Synology NAS:   df -h | grep '/volume'; cat /proc/mdstat; free -m; uptime
    NAS Auth-Log:   sudo grep -E 'Accepted|Failed|Invalid' /var/log/auth.log
    Docker:         sudo docker compose ps --format json
                    sudo docker compose logs --tail=5 app

  Voraussetzung:  SSH-Key des Windows-Rechners muss auf Zielgerät
                  in ~/.ssh/authorized_keys eingetragen sein.

  Genutzt von:    Check-SSH, Check-SynologyAPI, Check-Docker, Check-ExternalLogins

────────────────────────────────────────────────────────────────────────────────
  TECHNIK 5 – HTTP REST-API (POST JSON mit Session-Auth)
────────────────────────────────────────────────────────────────────────────────

  PowerShell-Cmdlet:  Invoke-RestMethod
  Protokoll:          HTTP, POST-Methode mit JSON-Body
  Authentifizierung:  Session-basiert (Passwort → SID → Header)

  Pi-hole v6 Authentifizierungsablauf:
    1. POST http://<IP>/api/auth
       Body: {"password": "passwort"}
       Antwort: {"session": {"sid": "abc123..."}}

    2. GET http://<IP>/api/stats/summary
       Header: "sid: abc123..."
       Antwort: JSON mit Statistiken

    3. DELETE http://<IP>/api/auth
       Header: "sid: abc123..."
       → Session beenden (sauberes Logout)

  Code-Muster:
    $authBody = @{ password = $passwort } | ConvertTo-Json -Compress
    $authResp = Invoke-RestMethod -Uri "http://$IP/api/auth" -Method Post
                  -Body $authBody -ContentType "application/json"
    $sid = $authResp.session.sid
    $headers = @{ sid = $sid }
    $summary = Invoke-RestMethod -Uri "http://$IP/api/stats/summary"
                  -Method Get -Headers $headers

  Genutzt von:    Check-PiholeAPI


────────────────────────────────────────────────────────────────────────────────
  TECHNIK 6 – HTTPS REST-API (POST JSON mit Token-Auth)
────────────────────────────────────────────────────────────────────────────────

  PowerShell-Cmdlet:  Invoke-RestMethod
  Protokoll:          HTTPS (selbstsigniertes Zertifikat → SkipCertificateCheck)
  Authentifizierung:  Token-basiert (Login-Command → Token → URL-Parameter)

  Reolink Kamera API-Ablauf:
    1. POST https://<IP>/api.cgi  (oder /cgi-bin/api.cgi)
       Body: [{"cmd":"Login","action":0,"param":{"User":{"userName":"admin","password":"pw"}}}]
       Antwort: [{"code":0,"value":{"Token":{"name":"abc123"}}}]

    2. POST https://<IP>/api.cgi?token=abc123
       Body: [{"cmd":"GetDevInfo","action":0,"param":{"channel":0}}]
       Antwort: Geräte-Informationen (Modell, Firmware)

    3. GET https://<IP>/cgi-bin/api.cgi?cmd=Snap&channel=0&token=abc123
       → JPEG-Snapshot (als Binärdaten)

    4. POST https://<IP>/api.cgi?token=abc123
       Body: [{"cmd":"Logout","action":0,"param":{}}]
       → Session beenden

  Lockout-Schutz:  3 Fehlversuche → 60 Minuten Sperrung (logs/camera_lockout.json)
  API-Endpunkt:    Automatische Suche zwischen /api.cgi und /cgi-bin/api.cgi

  Genutzt von:    Check-Camera (nur Reolink-Kameras)


────────────────────────────────────────────────────────────────────────────────
  TECHNIK 7 – HTTPS REST-API (POST Form-Encoded mit Basic Auth)
────────────────────────────────────────────────────────────────────────────────

  PowerShell-Cmdlet:  Invoke-RestMethod
  Protokoll:          HTTPS (selbstsigniertes Zertifikat)
  Authentifizierung:  HTTP Basic Auth (Authorization: Basic Base64(user:pass))
  Body-Format:        application/x-www-form-urlencoded

  MailStore Administration API:

  Auth-Header aufbauen:
    $authB64 = [Convert]::ToBase64String(
                  [Text.Encoding]::ASCII.GetBytes("admin:passwort"))
    $headers = @{ Authorization = "Basic $authB64" }

  API-Aufruf:
    $params = @{
        Uri                  = "https://192.168.80.120:8463/api/invoke/GetJobResults"
        Method               = "POST"
        Headers              = $headers
        SkipCertificateCheck = $true
        ContentType          = "application/x-www-form-urlencoded"
        Body                 = "fromIncluding=2026-04-27T14:00:00&toExcluding=2026-04-29T00:00:00&timeZoneId=W.%20Europe%20Standard%20Time"
    }
    $raw = Invoke-RestMethod @params

  BOM-Behandlung (PFLICHT bei MailStore):
    MailStore fügt UTF-8 BOM (0xFEFF) an den Anfang jeder Antwort.
    Invoke-RestMethod erkennt das und gibt einen String zurück statt ein Objekt.
    Lösung: $raw.TrimStart([char]0xFEFF) | ConvertFrom-Json

  URL-Encoding für Parameter:
    [Uri]::EscapeDataString($wert)
    Beispiel: "W. Europe Standard Time" → "W.%20Europe%20Standard%20Time"
    Wichtig: Datumsstring vorher als Variable aufbauen, nicht inline!

  Verwendete Endpunkte:
    /api/invoke/GetServerInfo    → Basis-Info (kein Body)
    /api/invoke/GetJobs          → Job-Liste mit Namen
    /api/invoke/GetJobResults    → Job-Ergebnisse (3 Pflichtparameter)
    /api/get-metadata            → API-Dokumentation (kein Body)

  Genutzt von:    Check-MailstoreAPI

────────────────────────────────────────────────────────────────────────────────
  U
────────────────────────────────────────────────────────────────────────────────

  [Uri]::EscapeDataString($string)
    URL-kodiert einen String (alle Sonderzeichen werden als %XX kodiert).
    Beispiel: "W. Europe Standard Time" → "W.%20Europe%20Standard%20Time"
    Wichtig: Leerzeichen → %20, / → %2F, : → %3A, + → %2B
    Verwendet für: MailStore API Parameter (fromIncluding, timeZoneId)

────────────────────────────────────────────────────────────────────────────────
  W
────────────────────────────────────────────────────────────────────────────────

  Wait-Job $job -Timeout $sekunden
    Wartet auf den Abschluss eines Hintergrundjobs.
    Gibt $null zurück wenn Timeout überschritten (Job läuft noch).
    Gibt Job-Objekt zurück wenn rechtzeitig fertig.

  Where-Object
    Filtert eine Kollektion von Objekten nach einer Bedingung.
    Kurzform: { $_.Eigenschaft -eq "Wert" }
    Reguläre Ausdrücke: -match "Muster"

  Write-Host
    Schreibt Text direkt auf die Konsole.
    -ForegroundColor  Textfarbe (Green, Yellow, Red, Cyan, Gray, ...)
    -NoNewline        Kein Zeilenumbruch am Ende
